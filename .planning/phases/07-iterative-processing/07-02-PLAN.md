# Plan 07-02: Chain Mode — Process Result Instead of Original

<objective>
Add a "Chain mode" toggle that makes the Process button use the current result as input instead of the original source image. Wire through GUI settings, worker pipeline, and CLI.
</objective>

<task id="3">
<title>Add chain_mode to WorkerMessage and pass result data</title>
<read_first>
- crates/prunr-app/src/gui/worker.rs (WorkerMessage enum)
- crates/prunr-app/src/gui/app.rs (process_items method)
</read_first>
<action>
Add to WorkerMessage::BatchProcess:
```rust
/// If Some, process this RGBA data instead of decoding from source_bytes.
chain_input: Option<Arc<image::RgbaImage>>,
```

In app.rs process_items, when building items Vec:
```rust
let chain = self.settings.chain_mode;
let items: Vec<(u64, Arc<Vec<u8>>, Option<Arc<image::RgbaImage>>)> = self.batch_items.iter()
    .filter(|i| filter(i) && !matches!(i.status, BatchStatus::Processing))
    .map(|i| {
        let chain_input = if chain { i.result_rgba.clone() } else { None };
        (i.id, i.source_bytes.clone(), chain_input)
    })
    .collect();
```

Update WorkerMessage construction to include chain_input.
</action>
<acceptance_criteria>
- WorkerMessage::BatchProcess has `chain_input: Option<Arc<image::RgbaImage>>`
- process_items passes result_rgba when chain_mode is true
- Items without a result (Pending status) pass None for chain_input even in chain mode
</acceptance_criteria>
</task>

<task id="4">
<title>Worker uses chain_input when provided</title>
<read_first>
- crates/prunr-app/src/gui/worker.rs (per-item processing in pool.scope)
</read_first>
<action>
In the per-item spawn, before the match on line_mode:

```rust
// Determine input: chain from previous result or decode from source
let input_image: Option<image::DynamicImage> = if let Some(ref rgba) = chain_input {
    Some(image::DynamicImage::ImageRgba8((**rgba).clone()))
} else {
    None
};
```

Then in each match arm:
- LinesOnly: use `input_image.unwrap_or_else(|| prunr_core::load_image_from_bytes(&img_bytes).unwrap())`
- AfterBgRemoval: if chain_input provided, skip bg removal and go straight to edge detection on the chain input
- Off (normal bg removal): if chain_input provided, encode to PNG bytes and pass to process_image_with_mask; otherwise use source_bytes as before

The key insight: process_image_with_mask takes `&[u8]` (encoded image bytes), so chain input needs to be encoded to PNG first. This is a one-time cost per process.
</action>
<acceptance_criteria>
- Worker processes chain_input when provided
- Worker falls back to source_bytes when chain_input is None
- All three LineMode arms handle chain_input correctly
- `cargo build` succeeds
</acceptance_criteria>
</task>

<action>
Add CLI flag:
```rust
/// Process the previous result instead of the original (for multi-pass scripting).

pub chain: bool,
```

Note: CLI chain mode only makes sense in interactive/scripting workflows where the user runs prunr multiple times on the same file. For single invocation it has no effect since there's no previous result. Document this in the help text.
